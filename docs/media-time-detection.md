# 如何判断媒体文件的拍摄时间

> 适用范围：照片（JPEG / HEIC / RAW / PNG 等）与视频（MP4 / MOV / MKV / AVI / MTS 等）。
> 目标：给出一个**多来源 → 优先级 → 一致性校验 → 兜底**的判定方法，并列出常见陷阱。
> 本文不涉及具体实现，只描述方法论。

---

## 一、判断思路总览

媒体文件的"拍摄时间"并不是单一字段，而是从多个候选来源中按可靠性排序、相互校验后得出的结论。

判定流程分四步：

1. **采集**：从文件中尽可能多地提取候选时间。
2. **过滤**：剔除明显错误的值（零值、未来时间、Epoch 占位等）。
3. **排序**：按来源可靠性排优先级，同优先级取较早值。
4. **校验**：用 GPS / 文件名 / 文件系统时间做交叉验证；冲突时告警而非沉默修正。

---

## 二、时间来源与可靠性分层

### P0 —— 容器内嵌"拍摄时刻"字段（最权威）

代表**设备在按下快门 / 开始录制那一瞬间写入文件本身的时间**。一旦写入，后续复制、移动、传输都不会改变。

**图片**

- EXIF `DateTimeOriginal`：按下快门的时间。
- EXIF `OffsetTimeOriginal`：对应的时区偏移（EXIF 2.31 之后才有）。
- 部分 RAW 厂商字段（Maker Note）中冗余记录的拍摄时间。

**视频**

- QuickTime / MP4：`com.apple.quicktime.creationdate`（含时区，iPhone 写入）。
- MKV / WebM：`DateUTC`（UTC，规范要求纳秒精度）。

### P1 —— 容器内嵌"数字化 / 写入"时间（次权威）

代表"图像被数字化"或"容器被写入"的时间。对数码相机而言通常等于 P0，但**扫描件、转码视频**会与 P0 不同。

**图片**

- EXIF `CreateDate` / `DateTimeDigitized`：图像被数字化的时间。
- XMP sidecar：`photoshop:DateCreated`、`exif:DateTimeOriginal` 的副本。

**视频**

- QuickTime atom `CreateDate`：以 1904-01-01 UTC 为零点的秒数（**注意：缺省值会写成 1904-01-01，必须当作空值剔除**）。
- MKV `DateUTC` 在转码后会被重写成转码时刻。

### P2 —— 文件名内嵌时间戳（启发式）

许多设备 / 应用以拍摄时间作为文件名的一部分，可靠性高但**格式不统一**，需要按设备来源识别：

- 主流相机命名：年月日 + 时分秒 / 序号。
- 主流手机命名：年月日 + 时分秒（前缀因厂商而异）。
- 截图命名：通常含完整年月日时分秒。
- 即时通信 / 网盘命名：常仅保留毫秒级 Unix 时间戳，**经常缺时分秒或时区**。

注意：从文件名提取的时间**没有时区信息**，需要根据来源（如 Pixel 写 UTC、iPhone 写本地时间）做经验性判断。

### P3 —— 旁路边信息（特殊场景才有）

- XMP sidecar 文件（`.xmp`）。
- 相册软件导出 JSON（如 Google Takeout 的 `photoTakenTime`）。
- 相册数据库（Apple Photos、Lightroom catalog）。

这些来源**权威性介于 P0 与 P2 之间**，但只有特定流程下才存在。

### P4 —— 文件系统时间（兜底，最不可靠）

- `mtime`（修改时间）：从手机直接拷出**通常**等于拍摄时间；但复制、解压、git checkout、rsync、编辑都会破坏它。
- `birthtime` / `crtime`（创建时间）：复制后变成"复制时刻"，**不是**拍摄时刻。
- `ctime`：inode 元数据变更时间，与拍摄无关。

经过 AirDrop、即时通信、网盘中转后，文件系统时间几乎一定失真，**只能作为最后兜底**。

## 三、优先级表（速查）

| 等级 | 来源类别 | 例子 | 时区 | 备注 |
|---|---|---|---|---|
| P0 | 容器内"拍摄时刻" | EXIF DateTimeOriginal、QuickTime creationdate、MKV DateUTC | 视字段 | 首选 |
| P1 | 容器内"数字化/写入" | EXIF CreateDate、MP4 atom CreateDate | UTC 或本地 | 注意 1904 占位 |
| P2 | 文件名解析 | 设备命名规则匹配 | 通常无 | 需按来源经验判断 |
| P3 | 旁路边信息 | XMP / Google Takeout JSON | 视来源 | 仅特定流程 |
| P4 | 文件系统时间 | mtime / birthtime | 本地 | 兜底 |

**同优先级冲突时**：取**较早**的值（更接近原始拍摄）。

---

## 四、时区处理

时区是这个问题里**最容易被忽视、也最容易出错**的部分。

- EXIF `DateTimeOriginal` 本质是**裸字符串**，没有时区。老照片无 `OffsetTime*` 字段时，无法精确转 UTC。
- QuickTime `creationdate` 含时区（iPhone 规范），但部分安卓厂商**违反规范**写本地时间。
- MKV `DateUTC`、MP4 atom `CreateDate` 是 UTC，但用户看到的"本地时间"取决于播放器解读。
- GPS 时间永远是 UTC，可作为**反推时区**的客观锚点：若 GPS UTC 与 EXIF 本地时间相差为整数小时（±少量秒），即为时区偏移。

**建议**：内部统一存 UTC + 时区偏移；展示时再按需转换。无时区信息时，用 GPS、文件名规则或用户设置的默认时区兜底，并显式标记"时区为推断值"。

## 五、常见陷阱

1. **1904 Epoch 占位**：未写入 `creation_time` 的 MP4 / MOV，解析库会返回 `1904-01-01`。**必须当作空值剔除**，否则会把整批视频判定到 1904 年。
2. **未来时间**：相机日期没设置正确会出现 2099 等年份。设硬上限到"当前时间 + 1 天"，超出即丢弃。
3. **过去时间下限**：摄影发明之前的年份（如 1900 之前）也应丢弃。实务上 1995 之前的数码照片几乎不可能存在，可作为软阈值。
4. **编辑后回退到 ModifyDate**：照片编辑会更新 `ModifyDate` / `DateTime`，但**不会**更新 `DateTimeOriginal`。**不要**在 P0 缺失时回退到 ModifyDate，它已经不是拍摄时间。
5. **HEIC / RAW 多重时间字段**：除了标准 EXIF，厂商 Maker Note 里还有冗余时间。两者不一致时以标准 EXIF 为准，但可保留冗余字段做校验。
6. **截图与屏幕录制**：往往**完全没有** EXIF，必须靠文件名或文件系统时间。
7. **即时通信压缩**：微信 / WhatsApp 等会**剥离 EXIF**，下载后只剩文件名里的毫秒时间戳。
8. **时区错配的批量误差**：iPhone 在跨时区旅行时，`creationdate` 含时区，但 EXIF `DateTimeOriginal` 仍是相机时间。同一批照片不同来源算出的时间可能相差数小时。
9. **GPS 时间 / 拍摄时间相差超过 24 小时**：通常说明相机日期错乱，应告警并以 GPS 时间为准。
10. **mtime 看似可靠的假象**：用户"刚刚从相机拷出来"的目录，mtime 看起来等于拍摄时间，但一次 `cp -r` 不带 `-p`、一次解压、一次同步盘扫描就会全部失效。**不要把 mtime 当 P0 用**。
11. **闰秒、夏令时**：跨夏令时切换日的时间需要库正确处理，自己拼字符串容易差 1 小时。

## 六、一致性校验建议

获得候选值后，做以下交叉校验：

- **P0 vs GPS UTC**：差值 > 24h，告警（相机日期错乱）。
- **P0 vs 文件名解析**：差值 > 1 天，告警（文件名可能来自其他源）。
- **P0 vs 文件系统 mtime**：差值 > N 天但 mtime < P0，仅提示（用户可能在调整时间）。
- **同一目录同来源批量**：P0 时间应单调或近似单调。出现孤立的"跳跃值"通常是数据问题。

校验冲突时**优先告警，避免静默修正**——错误的"修复"比错误的原始数据更难排查。

---

## 七、判定结果应包含什么

一个完整的"拍摄时间判定结果"建议包含以下字段，而非只给一个时间戳：

- **时间值**（UTC）
- **时区偏移**（若已知）
- **来源等级**（P0–P4）
- **来源字段名**（如 "EXIF DateTimeOriginal"）
- **置信度 / 是否推断**（如时区是推断得到）
- **冲突信息**（若有，列出所有候选值供人工复核）

这样上层应用（重命名、归档、相册）才能根据置信度决定是直接使用、提示用户确认，还是手工干预。

---

## 八、小结

> **结论**：拍摄时间不是单一字段，而是**多来源候选 + 优先级排序 + 一致性校验**的结果。
>
> 实务上：P0 容器内"拍摄时刻"为首选 → P1 容器内"数字化时间"次之 → P2 文件名启发式 → P3 旁路 JSON / sidecar → P4 文件系统时间兜底。每一层都要做范围合理性检查与时区处理，冲突时优先告警。
